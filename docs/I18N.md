# Localization Plan

Last updated: 2026-05-31

dayby ships in English, Japanese, and Korean for the first public release.

## Policy

- Keep `dayby` untranslated in every locale.
- Keep product copy short, calm, and low-pressure.
- Do not expose raw Supabase, network, or native-module English errors to users.
- Do not add native localization dependencies until the native rebuild batch is approved.
- Manual language choice is the source of truth. Device locale is only the initial guess.

## Supported Locales

- English: `en`
- Japanese: `ja`
- Korean: `ko`

The saved preference is `system | en | ja | ko`, stored as `dayby.localePreference`.

`system` is resolved at runtime. If device locale detection fails or returns an unsupported language, dayby falls back to English.

## Implementation

- Message catalogs live in `src/lib/i18n/messages.ts`.
- English is the source catalog.
- Japanese and Korean use `satisfies Messages`, so missing keys and plural-shape mismatches fail typecheck.
- Message keys are flat strings such as `home.heroTitle` and `post.error.signInRequired`.
- Plural values use `{ one, other }`. Plural keys require `{ count }`.
- Interpolation uses `{name}` placeholders with `string | number` values.

## Runtime

- `I18nProvider` reads the saved preference before the app renders.
- Splash is held until fonts and i18n preference resolution are ready.
- Storage failure still resolves `isReady = true` to avoid a blank app.
- `LanguageSwitcher` is available before and after sign-in.

## Errors

Services throw `I18nError(key, params)` for expected user-facing failures.

UI catch blocks call:

```ts
resolveErrorMessage(error, t)
```

`I18nError` becomes localized copy. Unknown raw errors become `common.unexpectedError`.

## Dates And Times

Display formatting is handled by locale-aware helpers from `src/lib/i18n/formatters.ts`.

Localized display targets:

- Home date
- Weekly range
- Monthly title
- Daily, weekly, and monthly time labels
- Archive delete dates

Do not change `src/lib/groupTime.ts` date calculation logic. Its `en-US` formatter is intentional because it creates stable numeric date parts for timezone math.

## Verification

Run before review:

```sh
npm run check:i18n
npm run typecheck
```

`check:i18n` checks:

- empty translated values
- plural-shape mismatches
- likely untranslated Japanese/Korean values that exactly match English

Allowed identical values are limited to brand names, sample codes, and format-only templates.
